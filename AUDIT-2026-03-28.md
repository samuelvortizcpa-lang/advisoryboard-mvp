## CRITICAL (3 findings)

### C1. OAuth state parameter is predictable — account linking attack
**Files:** `backend/app/services/google_auth_service.py:84`, `backend/app/api/integrations.py:169,226,281,336`
The OAuth `state` parameter is set to the plain Clerk `user_id` (e.g., `user_xxxx`). Combined with unauthenticated callback endpoints, an attacker who knows a victim's user_id can craft a callback URL to link the attacker's Google/Microsoft/Zoom/Front account to the victim's Callwen account. Classic OAuth CSRF.
**Fix:** Generate a cryptographic nonce, store in server-side session or signed cookie, validate in callback.

### C2. No vector index on `document_chunks.embedding` — all RAG searches are sequential scans
**File:** `backend/app/models/document_chunk.py:63`
The 1536-dim embedding column has no HNSW or IVFFlat index. Every RAG query does `ORDER BY cosine_distance()` as a full table scan across all chunks for a client. This is O(n) and will degrade as document volume grows. Ironically, `document_page_images.image_embedding` (which is marked "Legacy — unused") HAS an HNSW index.
**Fix:** Create an Alembic migration adding an HNSW index: `CREATE INDEX ix_chunk_embedding ON document_chunks USING hnsw (embedding vector_cosine_ops)`.

### C3. Stripe webhook returns 200 on processing errors — lost events
**Files:** `backend/app/api/stripe_routes.py:208-218`
When webhook processing throws an exception, the handler rolls back the DB (so the event is NOT recorded in `ProcessedWebhookEvent`) but returns `{"received": True}` (HTTP 200). Stripe sees success and won't retry. A `checkout.session.completed` that fails to provision the subscription = payment taken, no subscription created.
**Fix:** Return 500 on processing failure so Stripe retries. Or record the event ID in a separate transaction before processing.

## HIGH (8 findings)

### H1. OAuth callback endpoints have no authentication
**File:** `backend/app/api/integrations.py:169,226,281,336`
Google, Microsoft, Zoom, and Front callback endpoints accept `state` (user_id) with no verification the caller initiated the flow. Directly enables C1 above.

### H2. Next.js 14.2.35 has 7 known vulnerabilities
**File:** `frontend/package.json`
SSRF via Image Optimizer (GHSA-9g9p), HTTP request smuggling in rewrites (GHSA-ggv3), DoS via React Server Components (GHSA-h25m), unbounded image disk cache (GHSA-3x4c). Fix requires Next.js 15.5.14+ or 16.x.

### H3. Briefs router uses old auth pattern — bypasses org access control
**File:** `backend/app/api/briefs.py:33-41,74`
Uses `get_current_user` + `owner_id` instead of `get_auth` + `check_client_access`. Org members can't access briefs for clients assigned to them. No client access check at all.

### H4. Consent signing token brute-force — no rate limiting
**File:** `backend/app/api/consent_public.py:107-125`
Public endpoint to look up signing tokens has no rate limiting. Tokens can be enumerated.

### H5. No index on `organization_members.user_id` — hit on every authenticated request
**File:** `backend/app/models/organization_member.py:42`
`auth_context.py` resolves org membership on every request via this column. Sequential scan per request.

### H6. No indexes on `organization_members.org_id`, `client_access.client_id`, `client_access.user_id`
**Files:** `organization_member.py:35`, `client_access.py:37,44`
All used in JOINs and subqueries for access control. Missing indexes cause sequential scans.

### H7. N+1 query: `alerts_service.compute_alerts()` — 2 queries per client
**File:** `backend/app/services/alerts_service.py:133-164`
Loops through every client making 2 individual DB queries each (doc count + chat count). 100 clients = 200 queries.

### H8. Silent exception swallowing in document backfill
**File:** `backend/app/api/documents.py:226-227`
`except Exception: pass` hides all processing failures. No logging, no error response.
